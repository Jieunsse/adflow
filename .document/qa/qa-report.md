## BUG-001 · [verify] 배포 라이브 확인 시 항상 "의심" 판정 반환

**심각도**: 🔴 Critical

**[1] 상황:**

배포를 마친 후 배포가 실제로 라이브 됐는지 확인하기 위해 "방금 거 진짜 됐어?" 라고 입력했습니다.
SKILL이 동작하며 evidence를 수집하는 것처럼 보였지만, 최종 결과가 항상 아래와 같이 출력됩니다.

```
⚠️ 의심
- axhub status exit code 64
- state 필드 부재
- axhub logs exit code 64
```

실제 배포는 정상적으로 완료된 상태였고, 앱도 정상 동작 중이었습니다.\
어떤 앱을 대상으로 해도, 심지어 존재하지 않는 앱 ID를 입력해도 동일한 "의심" 판정이 나왔습니다.

**[2] 페인포인트:**

verify SKILL이 항상 "의심" 판정을 내리기 때문에 배포 성공 여부를 확인할 수 없습니다.\
"✅ 라이브 확정"을 한 번도 본 적이 없고, 실제로는 정상인 배포도 의심으로 분류되어 불필요한 불안감을 유발합니다.

**[3] 개선안:**

내부적으로 `axhub status`(인증·프로필 정보 반환) 명령어를 호출하고 있어 배포 상태를 담은 `state` 필드를 찾지 못하는 것이 원인입니다.\
배포 상태 확인에 필요한 명령어는 `axhub deploy status <deploy_id>` 이며, 해당 응답의 `status_name` 필드(`active` / `stopped` 등)를 기준으로 verdict를 판정해야 합니다.\
helper 내부 구현을 수정하면 정상적으로 ✅ / ⚠️ / ❌ 판정이 동작할 것으로 예상됩니다.

## BUG-002 · [trace] 최근 배포 이력 자동 조회 시 오류 발생

**심각도**: 🔴 High

**[1] 상황:**

배포 실패 후 "왜 실패했어?" 라고 입력했을 때 trace SKILL이 실행되었습니다.\
현재 세션에서 배포한 이력이 없거나, preflight가 `last_deploy_id`를 주입하지 못한 상황에서\
SKILL이 자동으로 최근 실패 배포를 찾으려 하다가 아래 오류와 함께 멈췄습니다.

```
exit 64: --app-id is required
```

**[2] 페인포인트:**

"왜 실패했어?"라고 물어봤는데 앱 정보를 직접 입력하라는 오류가 뜨는 상황이 당혹스럽습니다.\
배포를 방금 한 것도 아니고, 처음 사용하는 환경에서 이 문제가 특히 자주 발생할 것 같습니다.

**[3] 개선안:**

SKILL의 fallback 경로에서 `axhub-helpers list-deployments --limit 5 --json`을 호출할 때 `--app-id` 파라미터가 빠져 있습니다.\
preflight 응답에 포함된 `current_app` 값을 활용하여 `--app-id=$CURRENT_APP`을 자동으로 추가하면 해결됩니다.\
`current_app`도 없는 경우에는 "현재 앱을 특정할 수 없어요. 앱 슬러그를 알려주세요."처럼 명확한 안내가 있으면 좋겠습니다.

## BUG-003 · [trace] 존재하지 않는 배포 ID 입력 시 오류 미감지

**심각도**: 🔴 High

**[1] 상황:**

잘못된 배포 ID(예: `99999999`)를 입력하거나 타이핑 실수가 있는 경우에도\
SKILL이 오류 없이 정상 응답처럼 동작했습니다.\
출력 결과는 "실패 원인 없음" 형태였고, 실제로 배포 자체가 없는 건지 실패 이력이 없는 건지 구분할 수 없었습니다.

**[2] 페인포인트:**

잘못된 ID를 입력해도 오류 메시지 없이 조용히 "정보 없음"이 나오기 때문에\
"배포가 없는 건지", "ID가 틀린 건지", "실패 이력이 없는 건지" 세 가지를 구분할 방법이 없습니다.\
원인을 찾으려고 trace를 쓰는데 trace 결과가 신뢰되지 않으면 의미가 없습니다.

**[3] 개선안:**

helper가 존재하지 않는 deploy_id에 대해 유효한 배포와 동일한 빈 JSON을 반환하고 있어 발생하는 문제입니다.\
`"error":"not_found"` 필드 추가 또는 exit 65 반환으로 SKILL이 "해당 배포 ID를 찾을 수 없어요"라는 명확한 안내를 출력할 수 있도록 개선이 필요합니다.

## BUG-004 · [trace] 정상 완료된 배포 추적 시 안내 없이 종료

**심각도**: ⚠️ Medium

**[1] 상황:**

실패한 배포가 아닌, 정상적으로 완료된 배포 ID를 대상으로 "왜 실패했어?"를 입력했습니다.\
SKILL이 실행은 됐지만 failure 정보가 없어서인지 empathy card도 출력되지 않고,\
"실패 원인이 없다"는 안내도 없이 조용히 종료되었습니다.

**[2] 페인포인트:**

결과가 아무것도 나오지 않으면 SKILL이 동작하지 않은 건지, 아니면 진짜 실패 정보가 없는 건지 알 수 없습니다.\
특히 배포 ID를 직접 지정한 경우 "이 배포는 실패하지 않았어요"라는 확인이 있으면 훨씬 명확할 것 같습니다.

**[3] 개선안:**

`failure_reason:null`이고 `matched_patterns:[]`인 경우에 대한 early-exit 분기가 SKILL.md에 없습니다.\
해당 조건일 때 "이 배포는 실패 이력이 없어요. 배포 상태를 확인하려면 verify를 사용해보세요."처럼\
명확한 안내와 함께 관련 SKILL로 연결해주는 흐름이 있으면 좋겠습니다.

## BUG-005 · [install-cli] Homebrew 채널 선택 시 기존 CLI가 자동 업그레이드됨

**심각도**: ⚠️ Medium

**[1] 상황:**

CLI가 설치되어 있는 상태에서 install-cli SKILL을 통해 Homebrew 채널을 선택했을 때\
아래와 같은 메시지와 함께 업그레이드가 자동으로 시작되었습니다.

```
axhub 0.11.0 is already installed but outdated (so it will be upgraded).
==> Upgrading jocoding-ax-partners/tap/axhub
  0.11.0 -> 0.12.1
```

업그레이드 의사를 묻는 확인 절차 없이 바로 진행되었습니다.

**[2] 페인포인트:**

"설치"를 요청했는데 기존에 잘 쓰던 버전이 갑자기 올라가는 것은 예상치 못한 동작입니다.\
플러그인이 아직 0.12.1을 지원하지 않는다면 업그레이드 후 동작이 깨질 수 있고,\
업그레이드 사실 자체를 나중에 로그를 보고서야 알 수 있었습니다.

**[3] 개선안:**

Homebrew 채널 선택 후 `brew install` 실행 전에 `brew list axhub`로 기존 설치 버전을 확인하고,\
이미 설치된 경우 "현재 버전: 0.11.0이 설치되어 있어요. 0.12.1로 업그레이드할까요?"처럼\
명시적 업그레이드 동의를 받는 단계가 필요합니다.\
또한 플러그인 호환 버전 범위를 확인하여 호환되지 않는 버전으로의 업그레이드라면 사전에 경고해주면 좋겠습니다.

## BUG-006 · [routing-stats] confused 항목이 없을 때 빈 결과 출력

**심각도**: ⚠️ Low

**[1] 상황:**

"어떤 skill 많이 썼어?" 라고 입력하여 routing-stats를 실행했습니다.\
전체 통계는 정상 출력되었지만, "confused 보기" 옵션을 선택했을 때 아무 내용도 출력되지 않고 종료되었습니다.

**[2] 페인포인트:**

빈 결과가 나오면 기능이 동작하지 않은 건지, 정말 confused 항목이 없는 건지 알 수 없습니다.\
처음 사용하는 유저라면 버그라고 오해하기 쉬운 상황입니다.

**[3] 개선안:**

`confused_prompts:[]`인 경우 "지난 7일간 clarify가 발동된 항목이 없어요. 자연어 트리거가 잘 매칭되고 있는 것 같아요!"처럼\
빈 상태임을 명확히 안내하는 메시지가 출력되면 좋겠습니다.
